ADR-0022: Documentation Structure and Cross-References¶

Status¶

Accepted

Date¶

2025-02-27

Context¶

The Floxide framework's documentation needed to be more cohesive and better organized to help users understand and effectively use the framework. The documentation was spread across multiple locations with inconsistent cross-referencing and varying levels of detail.

Key issues that needed to be addressed: 1. Inconsistent documentation formats across different components 2. Missing or outdated architectural documentation 3. Lack of clear cross-references between related documents 4. Incomplete API documentation for some components 5. Examples that didn't fully demonstrate best practices

Decision¶

We have decided to implement a comprehensive documentation structure with the following components:

  1. API Documentation
  2. Detailed documentation for each crate
  3. Clear examples showing proper usage
  4. Links to relevant architectural docs

  5. Consistent format across all APIs

  6. Architecture Documentation

  7. Core Framework Abstractions
  8. Node Lifecycle Methods
  9. Event-Driven Workflow Pattern
  10. Batch Processing Implementation

  11. Async Runtime Selection

  12. Examples

  13. Basic workflow demonstrating core concepts
  14. Specialized examples for each pattern
  15. Error handling demonstrations

  16. Best practices implementation

  17. Cross-References

  18. Clear links between related documents
  19. References to relevant ADRs
  20. Links to example implementations
  21. References to external resources

Consequences¶

Positive¶

  1. Improved Usability
  2. Easier for new users to understand the framework
  3. Clear path from basic to advanced usage

  4. Better understanding of architectural decisions

  5. Better Maintainability

  6. Consistent documentation format
  7. Clear relationships between components

  8. Easier to identify missing documentation

  9. Enhanced Quality

  10. Examples demonstrate best practices
  11. Error handling is clearly documented
  12. Architecture decisions are well-explained

Negative¶

  1. Maintenance Overhead
  2. More documentation to keep updated
  3. Need to maintain cross-references

  4. Regular reviews required

  5. Potential Inconsistencies

  6. Risk of docs becoming outdated
  7. Need to coordinate updates across files
  8. More places to check when making changes

Implementation¶

The documentation has been updated to follow this structure:

  1. API Documentation
  2. Updated floxide-reactive.md
  3. Updated floxide-timer.md
  4. Updated floxide-core.md

  5. Added comprehensive examples

  6. Architecture Documentation

  7. Created/updated core architecture docs
  8. Added cross-references

  9. Improved consistency

  10. Examples

  11. Updated basic-workflow.md
  12. Added error handling examples

  13. Demonstrated best practices

  14. Cross-References

  15. Added links between related docs
  16. Referenced relevant ADRs
  17. Updated navigation structure
